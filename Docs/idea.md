# ConstraintLinkSetupTool （旧称: Tiloop Linker） アイデアと要件定義

## 1. 開発の背景と目的
VRChatにおいて、アバターに「義手」「サブアーム」「追加パーツ」を導入・改変する際、「ボーン構造を壊さずに正しい位置を維持してアニメーションを追従させる」という技術的課題があります。
Modular Avatar（MA）の `Setup Outfit` などで解決できない、「ボーンの配置・位置が一致しないケース」に対応するため、位置合わせを自動化せずユーザーの感性に任せ、**Rotation Constraintによる回転の同期と階層構造の構築（Parent Constraint）だけを自動化する**ツールを開発します。

---

## 2. コア・コンセプト
**「位置関係は設計者の感性、面倒な回転同期（コンストレイント設定）を一瞬で終わらせるツール」**

**対象ユーザー・ターゲットケース:**
- 義手・義足、サブアーム、浮্যুユニットなどのメカニカルパーツを装着したいユーザー。
- アバターの標準ボーンと１対１で対応させたいケースだけでなく、別のボーン（背中や腰など）を起点として追加パーツを追従させるケース。

**最大の強み:**
1. **自由なレイアウト:** アバターの腕の外側に「浮いている義手」や、背中から生える「サブアーム」など、位置合わせを気にせず自由に作れる。
2. **ボーン構造の柔軟性:** アバターの腕の長さと義手の長さが違っても、回転だけ同期するため動作が破綻しにくい。
3. **非破壊的:** アバター本体のボーン構造を書き換えないため、取り外しや修正が容易。

---

## 3. 自動化すべき「面倒な手順」とワークフロー

### 3.1. 階層構造の解析と複製（構造同期）
1. 義手側のルートボーン（起点）から末端（指先など）までを走査する。
2. ツールにアバター側の「追従起点ボーン（例：右肩、背中など）」を指定し、そこから先の子ボーンの階層順に、義手側のボーンを1対1でペアリングする。

### 3.2. コンストレイントの一括設定
1. 手作業だと数分〜数十分かかる、全ボーン（数十個に及ぶこともある指のボーン含む）に対する `RotationConstraint` コンポーネントの追加と `ConstraintSource`（追従先対象）の設定をワンクリックで実行。
2. オフセットの自動ロック:
   - 義手を配置した現在の位置・姿勢（オフセット）を維持したまま、自動で`Zero Out` (LockおよびActivate) 処理を行う。

### 3.3. 付け根（Base Bone）の位置固定
義手をアバターのRoot直下に配置した上で、義手のRoot（一番元の親ボーン）に対し `Parent Constraint` （または `Position` & `Rotation Constraint` ）を設定することで、アバター全体の動きに合わせて義手がついてくるようにする。

---

## 4. UI/UXの設計

### シンプルな設定画面
1. **Target Avatar Slot:** 改変したいアバター（または対象のルート）をドロップ。
2. **Prosthetic Object Slot:** 導入したい義手・追加パーツモデルをドロップ。
3. **Base Bone Assignment:** 義手の根元（例：肩）を、アバターのどのボーン（例：右肩、腰）に追従させるかを選択。
4. **Mapping Preview Display:**
   - ツールが自動推測した「アバターのボーン ↔ 義手のボーン」のマッピングリストを表示。
   - ユーザーが手動で微調整したり、追従させないボーンを除外できるようにする。
5. **Execute Button:** プロセスを実行する。

## 5. ボーン抽出・マッチングのロジックと実装方針

同プロジェクト内・手元の環境にある `BoneRendererSetupTool` のボーン名解析ユーティリティ (`BoneNameMatcher` および `HumanoidBonePatterns`) を活用し、堅牢なマッチングを実現する。

### 5.1. BoneNameMatcher の活用ポイント
- **正規化処理 (`NormalizeBoneName`):**
  - スペース、アンダースコア、ドット、数字（枝番）、`Bone_` プレフィックスを除去し、小文字化する。
  - 例: `UpperArm.L`, `upper_arm_L`, `Bone_UpperArm_L_01` など、表記揺れのあるボーン名を同一に扱えるようにする。
- **左右判定 (`HasSideIndicator`):**
  - 名称からの抽出（`left`, `right`, `_l`, `.r`など）により左右を判定。右腕コンストレイント時に左手ボーンを誤認する事故を防ぐ。
- **ヒューマノイド辞書マッピング (`TryMatchBone`):**
  - アバター側の `Animator` から取得した `HumanBodyBones` と、正規化された義手側のボーン名を辞書突き合わせにより確実に対応させる。

### 5.2. 「サブアーム」特有のマッピング（再帰的推測）
名前に依存しすぎるだけではサブアームのような「非標準構成（アバターに背中腕ボーンがない場合など）」の紐付けができないため、以下の２段構えとする。
1. **名前ベースの推測:** `BoneNameMatcher` による推測が成功した場合はそれを優先利用（明確な部位の一致）。
2. **階層ベースの推測:** ユーザーが指定した起点ボーン（Base Bone）をもとに、親子関係の「深さ（Depth）」や「インデックス」に基づいて機械的にマッピングする。これにより、名前が滅茶苦茶でも階層構造が合っていれば同期可能になる。

### 5.3. VRChat API との連携
- Unity標準の `RotationConstraint` に代えて、可能であればVRChat SDKの `VRCRotationConstraint` を使用する（パフォーマンスに優れるため）。
- スクリプト側で `#if VRC_SDK_VRCSDK3` などマクロやリフレクションを使用し、プロジェクトへのSDK導入状況に応じて適用するコンポーネントを切り替える。

## 6. 今後のステップ（フェーズ定義）

**フェーズ 1: UIと基本構造の構築**
- アバター、パーツ、起点ボーンを選択する `EditorWindow` の作成。
- プロジェクト内にある `BoneNameMatcher` のコードをリンク（アセンブリ参照）または移植。

**フェーズ 2: マッピングエンジンの実装**
- 階層順＋名前ベースのボーンペアリングを構築。
- 推測結果を UI 上でプレビュー（リストやツリー表示）し、ユーザーが編集できるようにする。

**フェーズ 3: コンストレイントエンジンの実装**
- パーツをアバター直下に Re-parent する機能。
- ルートに `ParentConstraint`、子ボーン群に `RotationConstraint` をバルクで付与し、かつ現在の Transform の姿勢（Offset）をロックして Activate する処理。

**フェーズ 4: リファインとリリース**
- 指のボーンを一括で処理から外す / 含めるなどの便利機能。
- Undo（取り消し）対応、エラーハンドリング。
